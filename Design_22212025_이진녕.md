## 1. Introduction

### 1) Executive Summary
본 시스템("오픈소스 몰 엔진")은 이커머스의 핵심인 **'데이터 파이프라인의 설계와 정책 최적화'**에 초점을 맞추어 기획 및 분석된 내용을 바탕으로 실제 소프트웨어 구현을 위한 구체적인 아키텍처와 객체 설계를 명세한다. 분석 단계에서 도출된 도메인 모델을 확장하여 Controller, Service, Repository 계층을 포함한 상세 Class Diagram과 흐름을 정의하며, 특히 Spring Framework 기반의 의존성 주입(DI)과 전략 패턴(Strategy Pattern)을 활용한 설계의 이점을 구체화한다.

### 2) Design Goals (설계 목표)
* **모듈화 및 관심사 분리(SoC):** UI/API 계층(Controller), 비즈니스 로직(Service), 데이터 접근(Repository)을 철저히 분리하여 유지보수성을 극대화한다.
* **유연한 정책 확장(OCP 준수):** `DiscountPolicy` 인터페이스와 `PolicyResolver`를 통해 기존 코드를 수정하지 않고도 새로운 할인 정책을 추가할 수 있는 아키텍처를 설계한다.
* **정합성 및 동시성 제어:** 다수의 동시 결제 요청에 따른 재고 부족(Oversell) 문제를 방지하기 위해 DB 수준의 비관적 락(Pessimistic Lock)을 `ProductRepository`에 적용하여 일관성을 보장한다.
* **정밀 연산 설계:** 모든 금전적 데이터 연산은 `BigDecimal`을 사용하여 부동소수점 오차를 원천적으로 차단한다.

## 2. Class Diagram

분석 단계의 도메인 모델을 바탕으로 시스템의 전반적인 구조(Controller - Service - Repository - Domain)를 표현한 상세 클래스 다이어그램이다.

```mermaid
classDiagram
    %% Presentation & Application Layer
    class OrderController {
        + createOrder(OrderRequestDto) ResponseEntity
    }
    class OrderService {
        - ProductRepository productRepository
        - PolicyResolver policyResolver
        - PaymentGateway pgClient
        - OrderRepository orderRepository
        + processOrder(Long customerId, List~ItemDto~ items, String idempotencyKey) OrderResponseDto
    }
    
    %% Domain Layer
    class PolicyResolver {
        - List~DiscountPolicy~ policies
        + applyDiscountRules(Customer, List~OrderItem~) BigDecimal
    }
    class DiscountPolicy {
        <<interface>>
        + getPriority() int
        + isExclusive() boolean
        + discount(Customer, OrderItem) BigDecimal
    }
    class RateDiscountPolicy {
        - BigDecimal discountRate
        + discount(Customer, OrderItem) BigDecimal
    }
    class FixDiscountPolicy {
        - BigDecimal discountAmount
        + discount(Customer, OrderItem) BigDecimal
    }
    
    class Product {
        - Long productId
        - String name
        - BigDecimal price
        - int stockQuantity
        + removeStock(int quantity)
        + addStock(int quantity)
    }
    class OrderItem {
        - Long orderItemId
        - Product product
        - int quantity
        - BigDecimal unitPrice
        - List~DiscountInfo~ appliedDiscounts
    }
    class Order {
        - Long orderId
        - String idempotencyKey
        - Customer customer
        - List~OrderItem~ orderItems
        - BigDecimal totalOriginalPrice
        - BigDecimal totalDiscountAmount
        - OrderStatus status
        + calculateFinalPrice() BigDecimal
        + changeStatus(OrderStatus status)
    }
    class Customer {
        - Long id
        - String name
        - Grade grade
        + getGrade() Grade
    }
    class OrderStatus {
        <<enumeration>>
        CREATED, PAYMENT_PENDING, COMPLETE, FAILED, CANCELED
    }
    
    %% Infrastructure Layer
    class ProductRepository {
        <<interface>>
        + findByIdWithPessimisticLock(Long id) Product
    }
    class OrderRepository {
        <<interface>>
        + save(Order order) Order
        + findByIdempotencyKey(String key) Optional~Order~
    }
    class PaymentGateway {
        <<interface>>
        + approvePayment(PaymentRequest) PaymentResponse
    }

    %% Relationships
    OrderController --> OrderService : invokes
    OrderService --> ProductRepository : uses
    OrderService --> OrderRepository : uses
    OrderService --> PolicyResolver : uses
    OrderService --> PaymentGateway : calls
    
    PolicyResolver o-- DiscountPolicy : aggregates
    DiscountPolicy <|.. RateDiscountPolicy : implements
    DiscountPolicy <|.. FixDiscountPolicy : implements
    
    OrderService ..> Order : creates
    Order *-- OrderItem : contains
    OrderItem --> Product : references
    Order --> Customer : placed by
    Order --> OrderStatus : has
```

### 설계 핵심 포인트
* **Repository 분리:** `ProductRepository`에 `findByIdWithPessimisticLock` 메서드를 명시하여 재고 접근 시 락(Lock)을 획득하도록 설계.
* **OrderService 주도:** 트랜잭션의 진입점으로, 멱등성 검증, 재고 차감, 할인 적용, 결제 승인 요청을 오케스트레이션(Orchestration)함.

## 3. Sequence Diagram

각 유즈케이스별 상세 호출 흐름 및 컴포넌트 간 상호작용을 표현한 시퀀스 다이어그램이다.

### 3.1 UC1. 상품 정보 및 실시간 재고 관리 (등록, 수정, 삭제)
```mermaid
sequenceDiagram
    actor Admin as Admin (운영자)
    participant Ctrl as ProductController
    participant Svc as ProductService
    participant Repo as ProductRepository
    participant P as Product (Entity)

    %% 상품 등록
    Admin->>Ctrl: POST /api/products (ProductRequestDto)
    Ctrl->>Svc: createProduct(dto)
    Svc->>P: new Product(name, price, stock)
    Svc->>Repo: save(product)
    Repo-->>Svc: Product 저장 완료
    Svc-->>Ctrl: Product 반환
    Ctrl-->>Admin: 201 Created (등록 정보)

    %% 상품 수정 (Optimistic Locking)
    Admin->>Ctrl: PUT /api/products/{productId} (ProductRequestDto)
    Ctrl->>Svc: updateProduct(id, dto)
    Svc->>Repo: findById(id)
    Repo-->>Svc: Product 반환
    Svc->>P: update(name, price, stock)
    Note right of P: @Version 필드로 낙관적 락 감지 준비
    Svc-->>Ctrl: 수정된 Product 반환
    Ctrl-->>Admin: 200 OK (수정 정보)

    %% 상품 삭제
    Admin->>Ctrl: DELETE /api/products/{productId}
    Ctrl->>Svc: deleteProduct(id)
    Svc->>Repo: existsById(id)
    alt 상품 미존재 시
        Repo-->>Svc: false
        Svc-->>Ctrl: IllegalArgumentException 예외 발생
        Ctrl-->>Admin: 400 Bad Request
    else 상품 존재 시
        Repo-->>Svc: true
        Svc->>Repo: deleteById(id)
        Svc-->>Ctrl: void
        Ctrl-->>Admin: 204 No Content
    end
```

### 3.4 UC4. 지능형 실시간 할인 금액 산출
```mermaid
sequenceDiagram
    participant Svc as OrderService
    participant Resolver as PolicyResolver
    participant Settings as DiscountPolicySettings
    participant Policy as DiscountPolicy (Rate/Fix)
    participant Item as OrderItem
    
    Svc->>Resolver: applyDiscountRules(Customer, List~OrderItem~)
    loop 각 OrderItem에 대해
        Resolver->>Policy: discount(Customer, OrderItem)
        Policy->>Settings: 런타임 활성화 여부 & 설정값 조회 (isRatePolicyActive, getRate 등)
        Settings-->>Policy: 설정 정보 반환
        
        alt 정책 활성화 상태
            Policy->>Policy: 등급별 할인 적용 및 금액 산출 (BigDecimal)
            Policy-->>Resolver: 할인 금액 반환
            Resolver->>Item: appliedDiscounts 리스트에 DiscountInfo 추가
        else 정책 비활성화 상태
            Policy-->>Resolver: BigDecimal.ZERO 반환
        end
    end
    
    Resolver->>Resolver: 우선순위(priority)와 단독 적용 여부(exclusive)에 따라 적용할 최종 할인 선별
    Resolver-->>Svc: 총 최종 할인 금액 반환
```

### 3.5 UC5. 결제 승인 및 데이터 무결성 검증 (주문 처리)
```mermaid
sequenceDiagram
    actor C as Customer
    participant Ctrl as OrderController
    participant Svc as OrderService
    participant PRepo as ProductRepository
    participant Resolver as PolicyResolver
    participant PG as PaymentGateway
    participant ORepo as OrderRepository

    C->>Ctrl: 1. POST /api/orders (idempotencyKey 포함)
    Ctrl->>Svc: 2. processOrder(Request)
    
    Svc->>ORepo: 3. findByIdempotencyKey(key)
    alt 중복 요청인 경우
        ORepo-->>Svc: 기존 Order 반환
        Svc-->>Ctrl: 4. 기존 처리 결과 반환
        Ctrl-->>C: 5. 결제 응답 (중복 차단)
    end
    
    Svc->>PRepo: 6. findByIdWithPessimisticLock (재고 락킹)
    PRepo-->>Svc: Product Entity 반환
    
    Svc->>Resolver: 7. applyDiscountRules (할인 연산 요청)
    Resolver-->>Svc: 총 할인 금액 및 내역 반환
    
    Svc->>Svc: 8. 최종 금액 산출 및 Order 객체 생성 (상태: PAYMENT_PENDING)
    Svc->>Product: 9. removeStock() (재고 선차감)
    Note right of Product: Optimistic Lock Version Check 포함
    
    Svc->>PG: 10. approvePayment (PG사 승인 요청)
    activate PG
    
    alt PG 승인 성공
        PG-->>Svc: 11. 승인 성공 응답
        Svc->>Order: 12. changeStatus(COMPLETE)
        Svc->>ORepo: 13. save(Order) (DB Commit)
        Svc-->>Ctrl: 14. 결제 완료 Response Dto
        Ctrl-->>C: 15. 200 OK (영수증)
    else PG 통신 실패 또는 잔액 부족
        PG-->>Svc: 11. 에러 응답 / Timeout
        Svc->>Product: 12. addStock() (재고 롤백)
        Svc->>Order: 13. changeStatus(FAILED)
        Svc->>ORepo: 14. save(Order) (실패 이력 저장)
        Svc-->>Ctrl: 15. 결제 실패 Exception
        Ctrl-->>C: 16. 400/500 Error (실패 사유)
    end
    deactivate PG
```

### 3.6 UC6. 주문 취소 및 재고 복구
```mermaid
sequenceDiagram
    actor C as Customer / Admin
    participant Ctrl as OrderController
    participant Svc as OrderService
    participant ORepo as OrderRepository
    participant PRepo as ProductRepository
    participant P as Product

    C->>Ctrl: POST /api/orders/{orderId}/cancel
    Ctrl->>Svc: cancelOrder(orderId)
    
    Svc->>ORepo: findById(orderId)
    ORepo-->>Svc: Order 반환
    
    alt 주문 상태가 COMPLETE가 아닌 경우
        Svc-->>Ctrl: IllegalStateException 발생 (취소 불가)
        Ctrl-->>C: 400 Bad Request
    else 주문 상태가 COMPLETE인 경우
        loop 주문의 각 OrderItem에 대해
            Svc->>PRepo: findByIdWithPessimisticLock(productId) (비관적 락 획득)
            PRepo-->>Svc: Product 반환
            Svc->>P: addStock(quantity) (재고 복구)
        end
        Svc->>Order: changeStatus(CANCELED)
        Svc->>ORepo: save(Order) (DB Commit)
        ORepo-->>Svc: Order 저장 완료
        Svc-->>Ctrl: OrderResponseDto 반환
        Ctrl-->>C: 200 OK (취소 성공 응답)
    end
```

## 4. State machine Diagram

주문(Order) 객체의 생명주기와 상태 변화를 표현한 State Machine Diagram이다. 이를 통해 결제 및 재고 처리 과정에서의 상태 전이 규칙을 명확히 한다.

```mermaid
stateDiagram-v2
    [*] --> CREATED : 고객이 장바구니에서 결제 요청
    
    CREATED --> PAYMENT_PENDING : 재고 선차감 및 할인 연산 완료
    
    PAYMENT_PENDING --> COMPLETE : PG사 결제 승인 성공
    PAYMENT_PENDING --> FAILED : PG사 승인 실패 (재고 롤백됨)
    
    COMPLETE --> CANCELED : 고객 결제 취소 요청 및 환불 완료
    FAILED --> [*]
    CANCELED --> [*]
    COMPLETE --> [*]
```

* **CREATED:** 주문 정보가 최초 수신되어 검증 중인 상태.
* **PAYMENT_PENDING:** 할인 산출이 끝나고 상품의 재고가 선차감(비관적 락 확보)되었으나 외부 PG 통신이 아직 완료되지 않은 대기 상태.
* **COMPLETE:** 시스템 내재화가 정상적으로 끝나고 데이터가 Commit된 최종 상태.
* **FAILED:** 한도 초과, 통신 에러 등으로 결제가 중단되어 보상 트랜잭션(재고 복구)이 실행된 상태.

## 6. Glossary

*   **Pessimistic Lock (비관적 락):** 트랜잭션 충돌이 발생할 것이라 가정하고 조회 시점부터 데이터베이스의 Row 단위 락을 걸어 정합성을 보장하는 기술.
*   **Layered Architecture (계층형 아키텍처):** 소프트웨어를 역할 단위(표현, 비즈니스, 데이터 계층 등)로 나누어 설계하는 아키텍처 패턴.
*   **Idempotency Key (멱등성 키):** 중복 결제 요청 방지를 위해 클라이언트가 생성하여 전달하는 고유 식별자.
*   **Compensation Transaction (보상 트랜잭션):** 분산 시스템이나 외부 연동(PG)에서 실패가 발생했을 때, 이전 단계에서 수행된 작업(예: 재고 선차감)을 원래 상태로 되돌리는 작업.

## 7. Implementation Alignment Addendum

본 보충 절은 실제 구현 코드와 설계 문서 사이의 추적성을 높이기 위해 추가한다. 기존 설계 문서가 결제 승인 흐름(UC5)을 중심으로 작성되어 있었으므로, 구현에서는 분석 문서의 UC1, UC2, UC3, 주문 취소 상태까지 포함하도록 API와 서비스를 확장하였다.

### 7.1 Use Case to Implementation Mapping

| Use Case | Implemented Component | Endpoint / Method | Notes |
| :--- | :--- | :--- | :--- |
| UC1. 상품 정보 및 실시간 재고 관리 | `ProductController`, `ProductService`, `ProductRepository`, `Product` | `POST /api/products`, `PUT /api/products/{productId}`, `DELETE /api/products/{productId}` | 상품 등록, 수정, 삭제를 지원한다. 상품 수정 충돌 감지를 위해 `Product`에 `@Version` 기반 optimistic locking을 적용한다. |
| UC2. 상품 검색 및 상세 조회 | `ProductController`, `ProductService` | `GET /api/products?keyword=...`, `GET /api/products/{productId}` | 전체 상품 조회, 이름 기반 검색, 단건 상세 조회를 지원한다. |

### 7.2 Locking Strategy Clarification

동시성 제어는 사용 시나리오에 따라 분리한다.

* **상품 관리 수정(UC1):** 운영자의 상품 정보 수정은 충돌 빈도가 낮으므로 `Product.version`의 `@Version`을 통한 optimistic locking을 사용한다.
* **결제 재고 차감(UC5):** 동일 상품에 대한 동시 결제 요청은 oversell 위험이 크므로 `ProductRepository.findByIdWithPessimisticLock()`에 `LockModeType.PESSIMISTIC_WRITE`를 적용한다.
* **주문 취소:** 재고 복구 역시 결제와 같은 재고 변경 흐름이므로 상품 row에 pessimistic lock을 획득한 뒤 `addStock()`을 수행한다.

### 7.3 Policy Configuration Model

할인 정책은 `DiscountPolicy` 인터페이스로 확장 가능하게 유지하면서, 기본 구현체 두 개를 제공한다.

* `RateDiscountPolicy`: VIP/VVIP 등급에 비율 할인 적용. 기본값은 10%.
* `FixDiscountPolicy`: 상품별 정액 할인 적용. 기본값은 1,000원이며 기본 설정은 exclusive이다.
* `DiscountPolicySettings`: 정책 활성화, 우선순위, exclusive 여부, 할인율, 정액 할인 금액을 보관하는 런타임 설정 컴포넌트이다.
* `PolicyController`: 운영자가 정책 설정을 조회하거나 변경할 수 있는 API를 제공한다.

구현 보강 후 다음 테스트로 설계 요구사항을 검증한다.

* `PolicyResolverTest`: 우선순위 기반 할인, exclusive 정책, 0원 하한 보정, 런타임 정책 비활성화 검증.
